Output: `AriClient` in `telephony/ari.py`; optional explicit `aiohttp>=3.14,<4` pin in pyproject; tests over a fake aiohttp session (REST) and a fake WS (event dispatch).
</objective>

    - A handler raising an exception is caught and logged; the loop continues (one bad event does not kill call control).
    - WS close / connection_lost causes run() to return cleanly (the entrypoint/controller decides on restart) — no infinite tight-loop.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| AriClient ↔ Asterisk ARI (REST + events WS) | Full call-control channel; credentials transit here |
| ARI event payloads → controller handlers | Untrusted event data (caller-influenced fields: caller ID, DTMF) |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-11-04-01 | Information disclosure | ARI Basic-Auth credentials | high | mitigate | Credentials held only in the aiohttp BasicAuth object; never logged, never in AriError messages (Task 1); sourced from env not TOML |
| T-11-04-02 | Denial of service | events dispatch loop | medium | mitigate | Handler exceptions + unknown event types are caught and logged; one bad event cannot kill call control (Task 2) |
| T-11-04-03 | Tampering | event payload fields | medium | mitigate | Controller (Plan 05) normalizes/validates caller-influenced fields; AriClient passes parsed dicts without trusting them for control decisions |
</threat_model>
