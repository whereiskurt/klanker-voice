<objective>
Put the OIDC sign-in gate in front of the mic (CLNT-08): tapping the attract CTA with no token starts an
authorization-code + PKCE redirect to auth.klankermaker.ai, the SPA callback route exchanges the code, and
the access token is held **in memory only** (D-04 — XSS-safer, re-auth on refresh). Authenticated
no-access-tier users get the exclusive/invite-only gate (D-13, the "need a code" moment moved here from
Phase 3 D-07) rather than a dead-end.

Purpose: The next demoable slice — a real sign-in. After this, the flow is attract -> sign in -> (no-access
gate OR authenticated, mic-ready). It also establishes tokenStore.getToken() as the single Bearer source the
live-connect plan (05-04) uses. Output: A working, secure PKCE auth flow + the no-access gate.
</objective>

a countdown, and honest failure handling — **so that** I experience the "whoa" of the demo and can cleanly
reconnect for another turn.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SPA ↔ auth.klankermaker.ai | authorization-code + PKCE redirect + token exchange over HTTPS |
| token in browser memory | the access token is a bearer credential to metered paid APIs via /api/offer |
| Vite bundle → public | anything compiled into the client is public |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-05-03-S | Spoofing | authorization-code flow | high | mitigate | PKCE S256 (code_challenge_method=S256) + a random `state` validated on callback prevent code interception/CSRF; public client, no secret to leak. |
| T-05-03-I | Information Disclosure | access token storage | high | mitigate | Token held in module-scope memory ONLY — never localStorage/sessionStorage/cookie (grep-gated), minimizing XSS exfiltration and persistence. Verifier/state in sessionStorage are non-secret and cleared on callback. |
| T-05-03-I | Information Disclosure | Vite bundle | high | mitigate | Only PUBLIC OIDC config (client_id/issuer/audience/redirect) is baked in; no client secret exists for a PKCE public client; .env.example documents this. |
| T-05-03-T | Tampering | token audience/scope | medium | mitigate | Authorize request pins scope 'voice' + audience https://voice.klankermaker.ai so the issued token matches exactly what auth.py validates; a mis-scoped token is rejected server-side. |
| T-05-03-E | Elevation of Privilege | no-access gate bypass | medium | mitigate | The gate is UX only; real enforcement is the server start_gate (Phase 4). The client decodes tier_id merely to show the right screen — it never grants access; the server is authoritative. |
| T-05-03-SC | Tampering | npm installs | low | accept | No new packages beyond 05-02's vetted set; crypto uses the built-in Web Crypto API (no jwt/oidc library added). |
</threat_model>
