### Gaps Summary

**Status: CLOSED.** One concrete, reproducible bug was found in the initial verification pass and has
since been fixed by TDD (this gap-closure pass, commit cfc7013):

**Gap (now fixed): the `login_required` safe-degrade breadcrumb clear was undone by shared callback wiring.**
`Callback.tsx`'s `login_required`/`interaction_required` branch correctly calls `clearReturningUser()`
(per A-04's design), but then calls the `onAuthenticated` prop — which is the *same* callback used by
the successful code-exchange path. In `App.tsx`, that shared handler (`handleAuthenticated`) used to
unconditionally call `markReturningUser()`. The net effect was: a device that gets `login_required` (e.g.
a truly signed-out visitor who previously signed in, or an expired session) had its breadcrumb
re-armed within the same tick it was cleared — so `kmv_returning` stayed `"1"`, and on this tab's next
fresh session (new tab or reload after the sessionStorage `silent_tried` guard naturally expires), the
device would attempt the silent bounce again, and again, indefinitely. This directly falsified half of
Success Criterion #2 ("a `login_required` ... clears the breadcrumb").

This was originally reproduced (not just inferred) with a temporary integration test exercising the real,
un-mocked `App` + `Callback` component wiring exactly as shipped — the existing test suite didn't
catch it only because `Callback.loginRequired.test.tsx` passes a bare `vi.fn()` in place of the real
`onAuthenticated` handler, so the App-level re-mark was invisible to it.

**Fix applied (this pass):** removed the unconditional `markReturningUser()` call — and its now-unused
`import { markReturningUser } from "./auth/returningStore"` — from `App.tsx`'s `handleAuthenticated`.
`Callback.tsx` was left untouched: it already marks the breadcrumb itself on the successful
code-exchange path (right after `setToken`), so App no longer needs to, and now correctly never
re-arms it on the login_required/interaction_required path. A permanent regression test,
`apps/voice/client/src/App.breadcrumb.test.tsx`, was added exercising the real App+Callback wiring
(not a stub): written and run against the unfixed code first (RED — reproduced the exact failure), then
GREEN after the fix, plus a guard test proving the successful `?code=` path still marks the breadcrumb.
Full client suite re-ran clean (17 files / 102 tests, 100 prior + 2 new) and `npm run build`
(`tsc --noEmit && vite build`) is clean. Fix committed as cfc7013.

Everything else in the phase — greeting rendering/manifest/drift-guard, greeting player + deferred
Live handoff (with a genuine behavioral state-transition test, not just presence), and the server-side
`greet_first` toggle — is code-complete and verified, either directly or via a passing behavioral test.
The two workstream test suites (170 server / 102 client) both re-ran clean.
