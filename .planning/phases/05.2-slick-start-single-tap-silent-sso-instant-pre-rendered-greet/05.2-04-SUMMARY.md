# Phase 05.2 Plan 04: Server greet_first config toggle + persona tweak Summary

**Added `PersonaConfig.greet_first` (default True), shipped `pipeline.toml` with it `false`, guarded server.py's `register_greet_first` call behind it, and rewrote concierge.md's "Opening move" so KPH stops re-introducing itself — closing the WebRTC double-greet.**

## Accomplishments
- `PersonaConfig` gained a `greet_first: bool = True` field; `load_config` reads `[persona] greet_first` from TOML, defaulting True when the key is absent (back-compat) (B-06)
- `pipeline.toml` sets `[persona] greet_first = false` — the shipped config disables the server-side opener now that the client plays a pre-rendered greeting clip on tap
- `server.py`'s `_run_session` now calls `register_greet_first(transport, worker, built.context)` only inside `if cfg.persona.greet_first:` — the console/local harness path (which calls `greet_now` directly, not `register_greet_first`) is untouched
- `prompts/concierge.md`'s "Opening move" section rewritten so KPH does not re-introduce itself on the user's first turn, since a spoken greeting is now played by the client before the LLM ever sees a turn (B-07)
- New `apps/voice/tests/test_greet_first_config.py` pins both the default-True-when-absent and reads-False-when-set behaviors

## Decisions Made
- Confirmed the guard's attribute path is `cfg.persona.greet_first` (the `cfg = load_config()` local already in scope in `_run_session`), not `built.config.persona.greet_first` as the plan's illustrative snippet suggested — `BuiltPipeline` (apps/voice/src/klanker_voice/pipeline.py) has no `.config` field, only `pipeline`/`context`/`user_aggregator`/`assistant_aggregator`/`rtvi`. The plan's own `<read_first>` and `<action>` explicitly flagged this as something to confirm before editing.
- Added a local `_persona_stub` autouse pytest fixture in the new test file (not shown in the plan's `<behavior>` code block) so `_MINIMAL`'s relative `prompt_path = "prompts/concierge.md"` resolves to a real file inside each test's `tmp_path` — required both for `load_config`'s existence check and because the second test's `.replace()` needs the literal relative-path substring to match (an absolute path there would silently no-op the `.replace()` and fail the second assertion).
