# Phase 05.2 Plan 03: Greeting player + play-on-tap with deferred Live handoff Summary

**Client-side pre-rendered KPH greeting that plays the instant the user taps (unlocking iOS audio) and holds the visible Live handoff until BOTH the clip ends AND the transport connects — closing the dead-air gap without ever overlapping the opener with live STT.**

## Accomplishments
- `greetingPlayer.ts` — `playRandomGreeting()` loads the 05.2-02 manifest (module-level cache), picks a random clip, plays it on a fresh `Audio` element, and resolves an `ended` promise on the clip's `ended`/`error` events and on a rejected `play()` — it never rejects or hangs; returns `null` when the manifest has no clips (B-05, T-05.2-03-D mitigation).
- `useVoiceSession.ts` — `start()` now plays the clip right after mic grant (same tap gesture that unlocks iOS audio) and before `beginConnect()`; the CONNECTED branch defers its `dispatch(event)` behind `greetingEndedRef.current.then(...)` so App mounts Live only once the opener has finished AND the transport is connected (self-loop guard, T-05.2-03-T mitigation).
- Three clip-release paths wired exactly as the revised plan required: on clip-end (its own `ended`), on `stop()` (manual teardown), and on the **pre-connect** `TRANSPORT_ERROR` branch (so a failed connect never leaves the greeting playing over the retry UI / `UdpBlockedWall`).
- Full client suite green: **100/100** tests, `npm run build` (tsc + vite) clean. Server suite re-run by coordinator: **170/170**.

## Decisions Made
- **Deferred only the dispatch, not the connected bookkeeping.** On CONNECTED, `wasConnectedRef`/`connectedAtRef`/`reportSuccess()`/retry-reset stay synchronous; only the reducer dispatch that flips the visible state is gated behind `greetingEndedRef`. This preserves 05-06's post-connect-drop-vs-pre-connect-failure classification while still holding Live until the clip ends.
- **Followed the revised plan's pre-connect TRANSPORT_ERROR wiring verbatim** — the clip stop was added to the branch at `useVoiceSession.ts:~139` (dispatch + `reportFailure()`), explicitly NOT the post-connect `wasConnectedRef` teardown path above it (out of scope; that path runs after clip-end already).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] greetingPlayer test: `function` (not arrow) Audio mock + per-test module reset**
- **Found during:** Task 1 (greetingPlayer.test.ts)
- **Issue:** The plan's verbatim test snippet used an **arrow function** for the `window.Audio` mock implementation; under vitest 4.1.9 / jsdom, `playRandomGreeting()` calls `new Audio(...)`, and arrow functions have no `[[Construct]]` internal slot → `TypeError: ... is not a constructor`. Separately, the two test cases share `greetingPlayer.ts`'s module-level `manifestCache`, so the first test's cached manifest would leak into the "no clips" case.
- **Fix:** Changed the mock to a named `function` expression, and added `vi.resetModules()` + a per-test dynamic `import("./greetingPlayer")` so each case gets a fresh module (fresh cache). The module under test is unchanged — this is a test-harness correctness fix only.
- **Files modified:** `apps/voice/client/src/greeting/greetingPlayer.test.ts`
- **Verification:** `npx vitest run src/greeting/greetingPlayer.test.ts` — 2/2 pass; `tsc --noEmit` clean.
- **Committed in:** `5a9a9c0` (Task 1 commit)

**2. [Rule 3 - Blocking] greeting handoff test: import `act` from @testing-library/react (not react)**
- **Found during:** Task 2 (useVoiceSession.greeting.test.ts)
- **Issue:** Importing `act` from `react` produced "The current testing environment is not configured to support act(...)" because `IS_REACT_ACT_ENVIRONMENT` was not set — Testing Library's own `act` re-export sets it.
- **Fix:** Imported `act`, `renderHook`, `waitFor` all from `@testing-library/react`.
- **Files modified:** `apps/voice/client/src/transport/useVoiceSession.greeting.test.ts`
- **Verification:** `npx vitest run src/transport/useVoiceSession.greeting.test.ts` — 2/2 pass.
- **Committed in:** `8d5b30e` (Task 2 commit)
