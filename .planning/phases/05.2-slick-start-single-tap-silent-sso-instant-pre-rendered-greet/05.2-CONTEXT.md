<domain>
## Phase Boundary

Make the voice session start feel like one slick tap. Today the start is clunky in two
separable ways: (1) a **double-tap** — the access token is in-memory only (D-04), so on
every fresh load `isAuthenticated` is false and the first tap is spent on a full-page
sign-in redirect; and (2) **dead air** — KPH only greets after WebRTC connects (~1–2s
after the tap), so there is silence and no signal that it is the user's turn.

Two independent workstreams compose into one experience:

- **Workstream A — Single tap (silent SSO on load):** a returning-user breadcrumb triggers
  one top-level `prompt=none` OIDC bounce during initial load, so a returning visitor is
  already authenticated when they tap. First tap → mic + connect, no visible sign-in bounce.
- **Workstream B — Instant greeting:** a small set of pre-rendered KPH greetings (rendered
  from the *configured* TTS voice), one picked at random per conversation, played the instant
  the tap gesture fires while WebRTC connects underneath; the server's greet-first is disabled
  on the WebRTC path so KPH does not greet twice.

**This phase is polish on the existing Phase-5 client + Phase-1 pipeline.** The OIDC issuer,
PKCE flow, in-memory tokenStore, `/api/offer` signaling, SmallWebRTC transport, orb/captions,
`register_greet_first`, and persona already exist. This phase adds: a boolean breadcrumb + silent
SSO trigger; a `prompt=none` option on the authorize URL; a `login_required` branch in Callback;
pre-rendered greeting clips + render script + drift guard; a greeting player + deferred Live
handoff; and a `greet_first` config toggle.

**Not in scope (non-goals):** changing the in-memory-token security posture beyond the minimal
boolean breadcrumb (no persisted access token — explicitly rejected); solving the mobile
speakerphone self-loop in general (parked; headphones remain the belt-and-suspenders — this
phase only avoids *adding* overlap between the greeting clip and live STT); varying greeting
wording via the LLM at runtime (clips are pre-rendered; variety comes from a small random set).
</domain>

### Workstream A — single tap
- **A-01 (breadcrumb):** `localStorage kmv_returning=1` set on a successful *interactive* sign-in
  (Callback code exchange) and on App's interactive-return path; cleared on explicit sign-out and
  on a silent `login_required`. Holds **no token** — only "this device has signed in before."
- **A-02 (loop guard):** `sessionStorage kmv_silent_tried=1` set before the silent navigation,
  checked on mount — so returning from `/callback` does not immediately re-trigger the silent
  attempt. Clears naturally when the tab/session ends.
- **A-03 (silent attempt):** on mount, if breadcrumb set AND `!isAuthenticated` AND not already
  tried this load → **one top-level** `window.location.assign` to the issuer authorize endpoint
  with `prompt=none`. **Top-level navigation only, never an iframe** — the auth cookie flows
  first-party so iOS Safari ITP does not block it.
- **A-04 (Callback two outcomes):** `?code=…` → existing PKCE exchange → token in memory →
  breadcrumb set → `replaceState("/")`. `?error=login_required` (or `interaction_required`) →
  clear breadcrumb, `replaceState("/")`, land on Attract signed-out, **no error UI** (expected
  "session expired" path).
- **A-05 (issuer gate — DONE):** Task 0 `prompt=none` feasibility gate is **VERIFIED GREEN**
  against the live issuer (2026-07-06): no-session `prompt=none` returns
  `303 → https://voice.klankermaker.ai/callback?error=login_required` (no login-page render).
  No new oidc-provider client config needed. Recorded as done — NOT a plan task.

### Workstream B — instant greeting
- **B-01 (voice coupling):** greeting clips MUST be rendered from the `voice_id` in
  `apps/voice/pipeline.toml` (currently `UgBBYS2sOqTuMpoF3BR0` "Mark"), model `eleven_flash_v2_5`,
  speed `1.1` — same as live TTS. Re-render on any voice swap.
- **B-02 (audio format):** clips are **MP3** (`mp3_44100_128`) for universal iOS Safari `<audio>`
  support — not opus.
- **B-03 (source of truth + render):** `greetings.source.json` lists the texts;
  `scripts/render_greetings.py` (run via `make -C apps/voice greetings`) reads voice_id from
  pipeline.toml, calls ElevenLabs, emits `greeting-N.mp3` + `greetings.manifest.json`
  = `{voiceId, model, clips:[{text,file}]}`. Rendered assets + manifest are committed (no
  build-time ElevenLabs secret).
- **B-04 (CI drift guard):** a pytest asserts `manifest.voiceId == pipeline.toml voice_id` (and
  manifest clips == source texts) — CI fails if someone swaps the voice without re-rendering.
  Pure string comparison, no ElevenLabs key in CI.
- **B-05 (playback + handoff):** on `useVoiceSession.start()`, after mic grant (same gesture that
  unlocks iOS audio), immediately play a random manifest clip on a dedicated `<audio>` element,
  then `beginConnect()` underneath. **Do not surface Live until BOTH the clip has ended AND the
  transport reached connected** (self-loop guard — greeting audio and live STT must not overlap).
  Stop/release the clip on clip-end, on `stop()`, and on any transport error.
- **B-06 (server toggle):** `PersonaConfig.greet_first` (default `True` for back-compat;
  `pipeline.toml` sets it `false`); `server.py` calls `register_greet_first(...)` only when true.
  The console/local path keeps its own greeting regardless (no client clip there).
- **B-07 (persona tweak):** `prompts/concierge.md` "Opening move" — the client plays the opener;
  when the user speaks their first turn KPH answers directly and does NOT re-introduce itself.

### Greeting copy (approved, 3 — written for the ear: no emojis/markdown/lists, "DEFCON dot run" spelled out)
1. "Hey! I'm KPH — a useful concierge assistant that sounds a lot like Kurt. Let's dig into some of my projects, feel free to interrupt, and just be yourself."
2. "Hey! I'm KPH — a concierge that sounds a lot like Kurt and knows his whole world. Let's dig into some of the projects. Feel free to interrupt me anytime, and just be yourself."
3. "Hey there — I'm KPH, your KlankerMaker concierge. Curious about Kurt, the platform, or DEFCON dot run? Just start talking, and cut me off anytime."

<success_criteria>
## Success Criteria (what must be TRUE)

1. A returning user (previously interactively signed in on this device) loads voice.klankermaker.ai,
   one silent top-level `prompt=none` SSO bounce completes during load, and their **first tap** goes
   straight to mic + connect — no sign-in redirect on the tap.
2. A signed-out user loads to Attract instantly and does no silent attempt; a `login_required` from
   the silent bounce clears the breadcrumb and lands signed-out with no error UI.
3. The instant a returning user taps, a random pre-rendered KPH greeting clip plays (unlocking iOS
   audio on the same gesture), masking the connect gap; the Live handoff waits until BOTH the clip
   has ended AND the transport is connected (no greeting/STT overlap).
4. Greeting clips are rendered from the `voice_id` configured in pipeline.toml (MP3,
   `eleven_flash_v2_5`); a CI drift guard fails if the clips were rendered from a different voice
   than currently shipped.
5. Server `greet_first` is disabled on the WebRTC path (client owns the opener) so KPH does not
   greet twice; the console/local path is unaffected.
6. No access token is ever persisted — Workstream A adds only a boolean `localStorage` breadcrumb
   plus a `sessionStorage` per-load loop guard.
</success_criteria>

## Deferred / Out of Scope

  follow-up to configure the issuer's interaction policy would be a separate task, not this phase.
