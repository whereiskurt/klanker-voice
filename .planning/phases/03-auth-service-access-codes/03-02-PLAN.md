<objective>
Make the access code a user types at login actually mean something: resolve it against a new
access_codes table (checking expiry and a unique-user redemption cap), carry the resolved tier across
the pre-user login request to the post-auth token via a short-lived login_intent bridge, and stamp it
onto the user's AuthProfile — counting redemptions by unique user, latest-wins on re-entry.

This is the access-code vertical slice: after this plan, the code a user enters changes the tier
recorded on their profile (which Plan 03 turns into a token claim). Defines the ElectroDB key
templates that Plan 04's kv CLI must match byte-for-byte.

Purpose: AUTH-03 (code -> tier, unknown/blank -> no-access, login always succeeds) and AUTH-04
(expiry + max-redemption limits). Implements D-04, D-05, D-06, D-07.
Output: four new ElectroDB entities, two new AuthProfile fields, code-resolution in /api/login, and
the login->token bridge in auth.ts.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> /api/login | Untrusted inviteCode string crosses; resolved server-side |
| login request -> magic-link callback | Resolved tier crosses two requests via login_intent (email-keyed) |
| access_codes table | Values may also be written by the kv CLI (Plan 04) — treat as an input boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-06 | Elevation of privilege | Redemption-count race lets a capped code over-redeem | high | mitigate | Unique-user count gates on a conditional CodeRedemption.create({code,userId}); the increment only runs when the create succeeds — concurrent duplicate logins by the same user cannot double-count (D-06, Task 3) |
| T-03-07 | Info disclosure | Access-code brute-force / enumeration at /api/login | medium | mitigate | Unknown and blank codes both resolve to no-access uniformly (no oracle); Altcha PoW (Plan 01) raises per-attempt cost; login response is uniform |
| T-03-08 | Tampering | login_intent tier tampering across the two requests | medium | mitigate | login_intent is server-written and email-keyed; the browser never carries the tier; short TTL bounds leakage; latest-wins overwrite prevents stale-intent replay |
| T-03-09 | Elevation of privilege | Stale/permanent tier stamp lets an old code keep access | medium | mitigate | Tier source is activeTierId overwritten every login via the bridge (latest-wins, D-05); the DEF CON per-user quotaTier is NOT used as the source |
| T-03-10 | Spoofing | kv-written code injecting control chars into ElectroDB keys | low | mitigate | Codes normalized (lowercase+trim) on read; explicit key templates bound the key shape; kv validates charset (Plan 04) |
</threat_model>
