<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator -> kv -> DynamoDB | kv writes raw items an untrusted user later redeems; codes become access grants |
| kv item keys <-> ElectroDB reads | A key-format mismatch silently hides or duplicates records |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-10 | Tampering | kv-written code injecting control chars / malformed keys into the electro table | medium | mitigate | kv validates+normalizes the code charset before write (Task 2); explicit templates bound the key shape; round-trip test proves compatibility (Task 3) |
| T-03-17 | Repudiation | Silent key mismatch makes a created code invisible to login (operator thinks a code exists that the app rejects) | high | mitigate | Round-trip key-compat test (Pitfall 1) is a blocking acceptance criterion; keys derived from the shared Plan-02 templates, not reverse-engineered |
| T-03-18 | Elevation of privilege | kv run with over-broad AWS credentials mutating unrelated tables | low | accept | Operator tool run by a trusted operator with scoped credentials; --table defaults to the electro table only; documented |
| T-03-SC | Tampering | go get of cobra + aws-sdk-go-v2 | low | mitigate | Package Legitimacy Audit (03-RESEARCH.md) — both already in km's go.mod, approved; no [ASSUMED]/[SUS]/[SLOP], no human checkpoint required |
</threat_model>
