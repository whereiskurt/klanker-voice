<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| voice client -> /token | Client requests a resource-audienced access token |
| access token -> voice service (Phase 4) | Token crosses to an offline validator that trusts only the JWKS |
| OIDC_JWKS (SSM) -> provider | Private signing key material loaded at boot |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-11 | Tampering | JWT algorithm confusion (alg=none / HS via the published public key) | high | mitigate | getResourceServerInfo pins jwt.sign.alg RS256 (asymmetric); Phase-4 PyJWT MUST pin algorithms=["RS256"] (recorded in the contract). No HS256 path exposed |
| T-03-12 | Spoofing | Audience confusion — a token minted for another RP accepted by the voice service | high | mitigate | Single voice client + defaultResource forces the voice audience; getResourceServerInfo sets aud explicitly; Phase-4 validates aud == pinned audience |
| T-03-13 | Tampering | Divergent/rotating JWKS across the Fargate fleet breaks or weakens validation | high | mitigate | configuration.jwks sourced from a single shared SSM JWK Set; keys stable across restarts and tasks; one consistent /jwks |
| T-03-14 | Info disclosure | OIDC signing key leaked via repo/logs | high | mitigate | OIDC_JWKS delivered as an SSM SecureString via valueFrom; SOPS-encrypted at rest; never committed or logged |
| T-03-15 | Elevation of privilege | Over-broad claims let the token grant more than tier gating | medium | mitigate | extraTokenClaims emits ONLY tier_id + group (D-01 thin token); limits are read from the tiers table at session start, not from the token |
| T-03-16 | Tampering | Open redirect via voice client redirect_uri | low | mitigate | oidc-provider validates redirect_uri against the registered voice client allowlist |
</threat_model>
