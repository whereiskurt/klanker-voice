# Quick Task 260706-q42: Fix reusable-workflow concurrency deadlock in deploy.yml Summary

Changed `deploy.yml`'s concurrency group from `${{ github.workflow }}-${{ github.ref }}` to a literal `deploy-ecs-${{ github.ref }}` so the reusable deploy job no longer collides with (and deadlocks against) its caller workflow's own concurrency group.

## What Was Done

`build-voice.yml` and `build-auth.yml` both `uses: ./.github/workflows/deploy.yml` as a reusable workflow. Inside a called (reusable) workflow, the `${{ github.workflow }}` context resolves to the **caller's** workflow name (e.g. "Build: Voice"), not "Deploy: ECS Update". Since deploy.yml's own concurrency block used `${{ github.workflow }}-${{ github.ref }}`, the called deploy job requested the exact same concurrency group the parent build run already held, and GitHub aborted the deploy job as a deadlock. This was empirically confirmed against workflow run 28827809814, which showed 1 job (build, succeeded) with an overall run conclusion of failure and no deploy check-run.

The fix gives `deploy.yml` a distinct literal group name (`deploy-ecs-${{ github.ref }}`) that cannot collide with any caller's `github.workflow`-derived group, while keeping `cancel-in-progress: false` unchanged so concurrent voice/auth deploys still serialize correctly onto the shared `ecs-task`/`ecs-service` terragrunt state.
